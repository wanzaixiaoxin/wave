# 造物运输大亨 — 设计研发管线流程 v1.0

> **本文档定位**：定义从设计文档到可执行代码的完整工作流。谁、在什么时候、产出什么、谁检查、怎么交接。  
> **适用角色**：主策划、执行策划、主程序、开发程序员

---

## 一、管线总览

```
主策划输出设计总纲
    │
    ▼
执行策划拆分 + 细化
    │
    ▼
┌─── 主策划评审 ───┐
│  通过？           │──❌不通过→ 退回执行策划修改
└────────┬─────────┘
         │✅ 通过
         ▼
分配给程序员
    │
    ▼
技术设计（程序员出技术方案）
    │
    ▼
┌─── 主程评审 ─────┐
│  方案可行？       │──❌不可行→ 退回修改或找主策划协商
└────────┬─────────┘
         │✅ 可行
         ▼
编码开发（含单元测试）
    │
    ▼
┌─── 测试验收 ─────┐
│  通过？           │──❌不通过→ 退回修复
└────────┬─────────┘
         │✅ 通过
         ▼
合入主分支 → 提测
```

### 角色职责一句话

| 角色 | 核心职责 | 产出物 | 决策权 |
|------|----------|--------|--------|
| 主策划 | 制定方向和标准，评审把关 | 设计总纲、评审意见 | 一票否决 |
| 执行策划 | 将总纲细化为可执行的文档 | 模块设计文档 xlsx/md | 设计细节决策 |
| 主程序 | 技术可行性判断，架构把控 | 技术方案评审意见 | 技术决策 |
| 开发程序员 | 按设计编码，保证质量 | 代码 + 单元测试 | 实现细节决策 |

## 二、阶段一：任务拆分（执行策划）

### 2.1 拆分原则

主策划输出 `主策划总纲.md` 后，执行策划按以下原则拆分：

```
① 每个独立模块拆成一份独立文档
② 文档之间不互相依赖（各自完整，可独立执行）
③ 每份文档的工作量控制在 3-5 个工作日
④ 超出 5 个工作日的模块继续拆分子模块
```

基于 `主策划总纲.md` 的拆分结果：

### 2.3 拆分文档的命名规范

```
EP-{策划代号}_{模块英文名}.{后缀}

EP = Execution Planner（执行策划）
策代号：A（养成）/ B（经济）/ C（挑战）/ D（UI）

示例：
```

## 三、阶段二：文档细化（执行策划）

### 3.1 每份细化文档必须包含的内容

```
┌──────────────────────────────────────────────┐
│  📄 文档头部                                  │
│  ├─ 模块名称                                   │
│  ├─ 关联的总纲章节（指向主策划总纲的具体节）      │
│  ├─ 作者 / 日期 / 版本号                       │
│  ├─ 依赖的其他模块文档（明确标注"需要 EP-B1 完成后才能开始"）│
│  └─ 预估工时（供排期用）                        │
├──────────────────────────────────────────────┤
│  📋 1. 需求概述                                │
│     用 3-5 句话概括这个模块的核心功能            │
├──────────────────────────────────────────────┤
│  🔢 2. 详细规则（这是核心内容）                  │
│     所有数值、条件、分支、边界情况必须写清楚       │
│     不允许出现"待定""后续决定"等模糊表述         │
│     必须包含：正常流程 + 边界情况 + 异常处理     │
├──────────────────────────────────────────────┤
│  📊 3. 配置表（如果涉及数值）                   │
│     以表格形式列出所有可配数值                    │
│     格式：xlsx 或 Markdown 表格                 │
├──────────────────────────────────────────────┤
│  🖼️ 4. 交互流程（如果涉及界面）                  │
│     用户操作路径：从入口到完成的每一步            │
│     必须包含：成功流程 + 失败流程 + 取消流程      │
├──────────────────────────────────────────────┤
│  ✅ 5. 验收标准                                 │
│     列出 5-10 条具体的"做完的标准"              │
│     每条标准必须可用"是/否"判定                  │
│     格式：Given-When-Then                      │
├──────────────────────────────────────────────┤
│  ⚠️ 6. 风险与开放问题                           │
│     设计过程中发现的不确定点                      │
│     每个问题标注：状态（待解决/已提交主策划）      │
└──────────────────────────────────────────────┘
```

### 3.2 细化标准示例（以 EP-A3_性格特质系统为例）

```
📄 EP-A3_性格特质系统.md

头部：
  模块：性格特质系统
  关联总纲：主策划总纲.md → 2.4 节
  作者：执行策划 A
  版本：v1.0
  依赖：无（独立模块）
  预估工时：2 日

1. 需求概述
  每辆车出厂时随机获得 1 个性格特质，特质提供被动增益。
  当前版本 6 种特质（5 普通 + 1 稀有）。
  后续版本可追加。

2. 详细规则
  2.1 特质获取
    • 每辆车出厂时获得 1 个特质
    • 概率表：
      勤快：17%  |  强壮：17%  |  精准：17%
      聪明：17%  |  招财：17%  |  幸运：15% (稀有)
    • 稀有特质触发时，有特殊抽卡特效（金色闪光）

  2.2 特质效果（详细数值）
    • 勤快：订单完成速度 ×(1 - 0.15) = ×0.85
    • 强壮：载货量 ×(1 + 0.20) = ×1.20
    • 精准：暴击率 +5%（暴击=收入×2）
    • 聪明：获得的经验 ×(1 + 0.20) = ×1.20
    • 招财：收入 ×(1 + 0.10) = ×1.10
    • 幸运：暴击时收入 ×3 而不是 ×2

  2.3 传承
    • 拆解车辆时，有 25% 概率将特质传给下一辆同车型新车
    • 如果已拥有该特质，叠加概率（不重复获得）

  2.4 边界情况
    • 如果玩家拆解了所有车，传承特质消失
    • 稀有特质传承概率同普通特质（25%）
    • 传承时触发金色特效提示

3. 配置表（EP-A3_TraitConfig.xlsx）
   trait_id | name | rarity | effect_type | effect_value | probability
   ─────────┼──────┼────────┼─────────────┼─────────────┼────────────
   quick    | 勤快 | normal | speed       | 0.85         | 0.17
   strong   | 强壮 | normal | cargo       | 1.20         | 0.17
   ...

4. 交互流程
  [造车完成] → [随机抽取特质] → [普通：蓝光特效] / [稀有：金光特效]
    → [弹出特质卡片] → [玩家点击确认] → [特质应用到车辆]

5. 验收标准
  Given 玩家造了一辆新车
  When 车辆出厂
  Then 车辆获得 1 个特质（必中）
  And 特质来自 [勤快, 强壮, 精准, 聪明, 招财, 幸运]

  Given 玩家拆解了一辆有特质的车
  When 拆解完成
  Then 有 25% 概率下一辆同车型新车获得该特质
  And 有 75% 概率特质消失

6. 风险与开放问题
  [已解决] 稀有特质概率 15% 是否太低？
    主策划确认：15% 目标就是稀有感，暂不调整。
  [待解决] 后续版本新增特质是否需要新旧存档兼容？
    已提交主策划，等待回复。
```

## 三、阶段三：主策划评审

### 3.1 评审触发条件

```
执行策划完成一份细化文档后，提交给主策划评审。

提交方式：在项目管理工具中创建 Review Request，附上文档链接
```

### 3.2 评审标准（Checklist）

```
主策划评审 checklist：

□ 1. 与总纲一致
   该文档的设计是否与主策划总纲的原始需求一致？
   如果有偏离，是否标注了"偏离原因"并合理？

□ 2. 完整性
   所有边界情况是否都已覆盖？
   是否有"待定/后续再定"的模糊表述？

□ 3. 可执行性
   文档交给程序员，程序员能否直接按文档编码？
   （不依赖执行策划的口头解释）

□ 4. 不互相矛盾
   与其他模块的文档是否有冲突？
   （例如：A 说"经验上限 100"，B 说"经验上限 200"）

□ 5. 验收标准可测量
   验收标准是否每条都可以用"是/否"判断？
   有没有"看起来不错""体验好"这种主观标准？

□ 6. 依赖关系清晰
   是否标注了依赖其他模块的部分？
   是否标注了"本模块完成后，其他模块需要对接的接口"？
```

### 3.3 评审结果

```
评审结果有三种：

✅ 通过（Approved）
  文档可以直接交给程序员。

🔄 有条件通过（Conditionally Approved）
  文档基本可用，但有几个小问题需要修改。
  执行策划修改后不需要再次评审，直接提交修改记录。

❌ 退回（Rejected）
  文档有重大问题，需要重新设计。
  主策划给出退回理由和修改方向。
  执行策划修改后重新提交评审。

目标：大部分评审应在 24 小时内完成。
```

### 3.4 评审时效要求

```
紧急度      主策划响应时间   执行策划修改时间
──────────────────────────────────────────
P0（阻塞）    4 小时内        当天
P1（重要）    24 小时内        2 天内
P2（常规）    48 小时内        5 天内
```

## 四、阶段四：程序技术设计

### 4.1 技术设计

通过主策划评审的文档交给主程序，主程序分配给具体程序员。

程序员需要产出一份**技术设计文档**，包含：

```
📄 TD-{模块名}_技术设计.md

1. 数据结构
   - 定义该模块需要用到的 interface / class
   - 明确数据结构属于"运行时"还是"配置表"

2. 接口定义
   - 该模块对外暴露的 API
   - 该模块依赖的其他模块的 API

3. 状态机 / 流程图（如果涉及复杂状态）
   - 例如：订单的状态转换（等待→进行中→完成→结算）

4. 存储方案
   - 哪些数据需要存档？
   - 存档的 key 和 value 格式？

5. 测试方案
   - 核心逻辑的单元测试用例
   - 边界情况的测试场景
```

### 4.2 技术评审

```
主程序评审技术设计文档：

✅ 架构合理
   - 不与现有架构冲突
   - 遵循单向数据流原则
   - 模块间解耦

✅ 性能达标
   - 时间复杂度可接受
   - 不会引起内存泄漏
   - 没有不必要的频繁计算

✅ 可测试
   - 核心逻辑可以写单元测试
   - 外部依赖可以 mock

✅ 向后兼容
   - 不影响已有功能
   - 存档兼容
```

### 4.3 评审结果

```
通过          → 进入编码开发
需修改        → 程序员修改后再次提交主程评审
方案不可行     → 程序员 + 主策划 + 主程三方协商修改设计
                （如果技术实现成本过高，可能需要调整设计）
```

## 五、阶段五：编码与验收

### 5.1 编码规范

```
所有代码遵循以下规范：
  • TypeScript strict 模式
  • 命名规范：camelCase（变量/函数），PascalCase（类/接口）
  • 文件命名：kebab-case.ts
  • 所有函数必须有 JSDoc
  • 不允许硬编码数值（必须读配置表）
  • 提交前跑 eslint + prettier
```

### 5.2 单元测试要求

```
• 核心逻辑覆盖率 ≥ 80%
• 每个模块至少覆盖：
  - 正常流程 × 1
  - 边界情况 × 至少 2
  - 异常输入 × 至少 1
• 测试不依赖 UI（纯逻辑测试）
• 测试不依赖存档（使用 mock 数据）
```

### 5.3 验收流程

```
程序员完成开发 + 单元测试后，提交流程：

① 自测（程序员自己跑一遍）
  → 对照执行策划文档的"验收标准"逐条执行
  → 截图/录屏作为证据

② 主程验收（代码角度）
  → Code Review
  → 检查：代码质量、架构一致性、性能、测试覆盖

③ 主策划验收（功能角度）
  → 对照验收标准逐条执行
  → 检查：功能是否正确、交互是否符合设计

④ 测试组验收
  → 黑盒测试 + 边界测试 + 回归测试
  → 输出测试报告
```

### 5.4 验收状态

```
状态              含义
────────────────────────────────────
设计评审中         执行策划正在细化文档
主策划评审中        文档在主策划手上
程序技术设计中      程序员正在写技术方案
主程评审中          技术方案在主程手上
编码开发中          程序员正在写代码
主程验收中          Code Review 中
主策划验收中        功能验收中
测试中              测试组测试中
✅ 已完成           全部通过，合入主分支
❌ 已废弃           该设计被取消或替换
```

## 六、紧急通道

### 6.1 什么情况下走紧急通道

```
① 线上 bug 需要立即修复
② 某个模块阻塞了其他 3 个以上模块的进度
③ 主策划判断为"不做就来不及上线"的功能
```

### 6.2 紧急通道流程

```
正常流程：设计 → 评审 → 技术设计 → 评审 → 编码 → 验收
紧急通道：设计（简化版）→ 编码 → 补验收

区别：
  • 设计文档可以简化到"需求概述 + 核心规则"（2 页以内）
  • 跳过主策划评审，直接由主策划口头确认
  • 技术设计可以省略，直接编码
  • 测试可以后补，但合入前必须通过自动化测试

紧急通道使用后，必须在 3 个工作日内补全正常流程的文档。
```
