## 一、技术选型决策

### 1.1 选项对比

| 方案 | 语言 | 包体基准 | 2D 能力 | 低端机性能 | 社区资源 | 团队门槛 | **综合评分** |
|------|------|----------|---------|-----------|---------|---------|------------|
| **Cocos Creator** | TypeScript | 8-15MB | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | **4.7/5** |
| Unity + 2D | C# | 25-40MB | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | 3.8/5 |
| Godot 4 | GDScript | 15-25MB | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ | 3.5/5 |
| Phaser.js (H5) | TypeScript | 5-10MB | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 3.5/5 |
| Flutter + Flame | Dart | 12-20MB | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐ | 3.0/5 |

### 1.2 最终决策：Cocos Creator 3.x

```
选择 Cocos Creator 的理由（按重要性排序）：

① 2D 原生基因
  Cocos Creator 从诞生就是为 2D 手游设计的。
  Unity 的 2D 是 3D 引擎的"附加模式"，
  Cocos 的 2D 渲染管线是为 2D 精灵优化的。
  → 这意味着更少的 draw call、更高的帧率、更低的功耗。

② 包体控制
  一个 Cocos Creator 2D 游戏的空包约 8-12MB。
  同样功能的 Unity 2D 游戏空包约 25-35MB。
  我们的目标 < 50MB，Cocos 给美术资源留了 40MB 空间，
  Unity 只留了 15MB。
  → 意味着 Cocos 可以塞更多车辆动画/贴图。

③ 低端机性能
  Cocos Creator 在 Android 千元机上的 2D 渲染表现
  显著优于 Unity（基于多个已上线项目的实测数据）。
  → 覆盖更多用户，尤其是在目标市场的下沉设备。

④ 编辑器适配工作流
  Cocos Creator 的编辑器 UI 系统自带"对齐/布局/适配"，
  对于大量卡片式 UI（本游戏的核心交互）开发效率很高。
  → 减少 UI 联调时间。

⑤ 小游戏/Web 备选
  Cocos Creator 一键发布到微信小游戏、H5、抖音小游戏。
  如果后续需要扩展渠道，不需要重写代码。
  → 为未来的发行策略留了后路。
```

### 1.3 备选方案：如果团队没有 Cocos 经验

```
如果团队只有 Unity 经验，备选方案是 Unity + URP 2D Renderer。

代价评估：
  • 包体增加约 15-20MB（需要压缩纹理和音频来弥补）
  • 低端机 60fps 需要更严格的 draw call 控制
  • 开发效率不变（如果团队熟悉的话）
  • 后续不能发小游戏

决策原则：优先选团队熟悉的引擎。工具是手段，不是目的。
```

## 二、整体架构设计

### 2.1 架构分层

```
┌──────────────────────────────────────────────────────────┐
│                     表现层（View）                         │
│   SceneMgr  │  UIMgr  │  AnimationMgr  │  AudioMgr      │
│   (场景管理)  (UI管理)  (动画管理)         (音频管理)       │
├──────────────────────────────────────────────────────────┤
│                     逻辑层（Logic）                        │
│   VehicleSys │  OrderSys  │  FactorySys  │  TechSys      │
│   EventSys   │  AchieveSys│  ChallengeSys│  PrestigeSys  │
├──────────────────────────────────────────────────────────┤
│                     数据层（Data）                         │
│   GameModel  │  SaveMgr  │  ConfigMgr  │  StatsMgr      │
│   (运行时数据) (存档)      (配置表)        (统计)         │
├──────────────────────────────────────────────────────────┤
│                     基础设施（Infrastructure）              │
│   EventBus   │  TimerMgr │  PoolMgr    │  Logger        │
│   (事件总线)   (定时器)    (对象池)       (日志)          │
└──────────────────────────────────────────────────────────┘
```

### 2.2 各层职责（严禁越层）

```
表现层（View）
  • 负责所有渲染和用户输入
  • 只调用 Logic 层的接口，不直接修改 Data
  • 不包含任何游戏规则判断
  • 示例：点击"派单"按钮 → 调用 OrderSys.assignOrder()
    ✅ 正确：View 只负责"接到点击事件，调用逻辑接口"
    ❌ 错误：View 自己判断"订单是否可以接"

逻辑层（Logic）
  • 包含所有游戏规则和策略
  • 读 Data，写 Data，驱动 View 更新
  • 不依赖具体的 UI 组件或渲染方式
  • 示例：OrderSys.assignOrder() → 计算收入 → 写入 GameModel
    → 发送 EventBus.emit('order:assigned') → View 收到事件后更新 UI

数据层（Data）
  • 运行时状态的单一数据源
  • 所有游戏状态集中管理
  • 存档时序列化整个 GameModel

基础设施（Infrastructure）
  • 跨层共享的工具类
  • 不包含游戏逻辑
```

### 2.3 数据流方向

```
用户输入（触摸/点击）
    ↓
View（接收输入，调用 Logic 接口）
    ↓
Logic（处理规则，修改 Data）
    ↓
Data（状态变更，触发事件）
    ↓
EventBus（广播事件）
    ↓
View（收到事件，更新 UI）
    ↓
用户看到变化
```

**单向数据流，禁止反向调用。** Logic 层不直接操作 View 层的组件。

## 三、模块划分

### 3.1 核心模块清单

| 模块 | 文件路径（建议） | 核心类/接口 | 依赖 |
|------|-----------------|-------------|------|
| 车辆系统 | `logic/vehicle/` | VehicleSys, Vehicle | ConfigMgr, EventBus |
| 订单系统 | `logic/order/` | OrderSys, Order | VehicleSys, EventBus |
| 生产系统 | `logic/factory/` | FactorySys, ProductionLine | ConfigMgr, EventBus |
| 科技树 | `logic/tech/` | TechSys | EventBus |
| 随机事件 | `logic/event/` | RandomEventSys | EventBus |
| 成就系统 | `logic/achievement/` | AchievementSys | EventBus, StatsMgr |
| 挑战模式 | `logic/challenge/` | ChallengeSys, SpeedRush, Survival, Randomizer | 全部下层系统 |
| 轮回系统 | `logic/prestige/` | PrestigeSys | SaveMgr |
| 亲密度 | `logic/intimacy/` | IntimacySys | VehicleSys |
| 经济计算 | `logic/economy/` | EconomyCalc | ConfigMgr |

### 3.2 模块间依赖原则

```
严格单向依赖：
  VehicleSys → ConfigMgr
  OrderSys → VehicleSys
  FactorySys → ConfigMgr
  EconomyCalc → ConfigMgr
  TechSys → EconomyCalc
  
禁止循环依赖：
  ❌ VehicleSys ↔ OrderSys（互相引用）
  ✅ 通过 EventBus 解耦：VehicleSys 发射事件，OrderSys 监听

依赖注入：
  所有模块通过构造函数接收依赖，不直接 import 单例（方便测试和 mock）
```

## 四、核心时序设计

### 4.1 游戏启动时序

```
App.start()
  │
  ├→ LoaderMgr.init()          // 加载资源（显示加载界面）
  │    ├→ 加载配置表（VehicleConfig / TechConfig / ...）
  │    ├→ 加载 UI 预制体
  │    └→ 加载精灵/纹理
  │
  ├→ SaveMgr.load()             // 读取存档
  │    ├→ 有存档 → 恢复 GameModel
  │    ├→ 无存档 → 创建初始 GameModel
  │    └→ 计算离线收益（如果离线时间 > 10s）
  │
  ├→ LogicMgr.init()            // 初始化所有逻辑模块
  │    ├→ VehicleSys.init(GameModel.vehicles)
  │    ├→ OrderSys.init()
  │    ├→ FactorySys.init(GameModel.factory)
  │    ├→ TechSys.init(GameModel.techTree)
  │    └→ EventSys.init()
  │
  ├→ UIMgr.init()               // 构建 UI
  │    ├→ 创建 TopBar
  │    ├→ 创建 GarageScene
  │    ├→ 创建 BottomTab
  │    └→ 绑定事件监听
  │
  └→ GameLoop.start()           // 启动游戏循环
       ├→ EconomyTick(1Hz)
       └→ RenderLoop(60fps)
```

### 4.2 游戏循环（Economy Tick）

```
每 1 秒执行一次 EconomyTick（由 TimerMgr 驱动）

EconomyTick():
  1. FactorySys.tick()
     → 遍历所有产线，减少剩余时间
     → 有完成的 → 调用 VehicleSys.createVehicle()
     → 检查队列 → 开始下一个生产

  2. OrderSys.tick()
     → 检查进行中的订单，减少剩余时间
     → 完成的订单 → 计算收入 → 加入金币
     → 车辆获得经验 → 检查是否升级
     → 车辆获得亲密度

  3. EventSys.tick()
     → 概率判定是否触发随机事件
     → 触发 → 应用事件效果（临时 buff）
     → 检查事件到期 → 移除效果

  4. AchievementSys.check()
     → 遍历所有未解锁成就
     → 条件满足 → 解锁 → 发放奖励
     → EventBus.emit('achievement:unlocked')

  5. AutoSave()
     → 每 30 秒自动保存一次
```

### 4.3 离线收益计算时序

```
玩家打开 App
    │
    ├→ SaveMgr.calculateOfflineEarnings(saveData, offlineSeconds)
    │    ├→ 限定 maxOffline = 7200s (2h)
    │    ├→ 效率系数 = 0.4
    │    ├→ 模拟生产：
    │    │    各产线按离线前配置持续生产
    │    │    生产完成的车自动出售
    │    │  ← 不模拟订单（无经验/无亲密度/无随机事件）
    │    └→ 返回：{ gold, parts, carsProduced }
    │
    ├→ GameModel.applyOfflineEarnings(result)
    │
    └→ UIMgr.showOfflineDialog(result)
         └→ 显示："你不在的 X 小时里，产线共生产了 N 辆车..."
```

## 五、性能策略

### 5.1 渲染性能

```
目标：60fps，最低不低 30fps

具体措施：
① 精灵合批
   所有车辆卡片使用同一个 atlas（纹理图集）
   所有 UI 图标使用同一个 atlas
   → 减少 draw call 到 < 30

② 对象池
   车辆卡片、飘字、粒子等频繁创建销毁的对象使用对象池
   → 避免 GC 抖动

③ 视口裁剪
   车库中不在屏幕范围内的车辆卡片不渲染
   → 车辆超过 12 辆时仍然保持 60fps

④ 动画性能
   优先使用 Cocos 的 Tween 动画（基于属性插值，无额外 draw call）
   其次使用 spritesheet（2D 序列帧）
   避免使用骨骼动画（2D 游戏不需要）
```

### 5.2 内存策略

```
目标：运行内存 < 100MB

① 纹理管理
   所有纹理最大 2048×2048
   使用压缩纹理格式（Android: ETC2, iOS: PVRTC）
   不使用的场景纹理释放（车库↔工厂切换时释放另一套）

② 音频管理
   背景音乐（BGM）使用流式加载，不预加载到内存
   音效预加载到内存（< 10MB 总量）

③ 缓存策略
   配置表启动时全量加载到内存（< 200KB JSON）
   存档每次保存全量写入，不维护增量日志
```

### 5.3 启动速度

```
目标：从点击图标到可交互 < 5 秒

时间分配：
  引擎初始化：~1.5s
  资源加载：~2.0s（异步，显示加载界面）
  存档读取+离线计算：~0.5s
  UI 构建：~0.5s
  首帧渲染：~0.3s
  额外缓冲：~0.2s

优化手段：
  • 首屏只加载车库场景资源，工厂/殿堂资源后台延迟加载
  • 配置表合并为一个 JSON 文件，减少网络请求
  • 使用 Cocos 的 Asset Bundle 分场景打包
```

## 六、构建与发布

### 6.1 构建管线

```
开发阶段：
  源码 → TypeScript Compiler → 游戏包
  Cocos Creator 编辑器 → 预览（热更新）

发布阶段：
  Cocos Creator → Build
    ├─→ iOS: Xcode Archive → App Store
    ├─→ Android: Gradle Build → APK/AAB → 应用商店
    └─→ (备选) 微信小游戏: 微信开发者工具 → 上传
```

### 6.2 包体控制目标

```
总包体目标：< 50MB

分配预算：
  ├─ 引擎 & 框架代码：     ~12MB
  ├─ UI 资源和图集：        ~8MB
  ├─ 车辆设计资源（20辆）： ~12MB
  ├─ 背景和场景资源：       ~5MB
  ├─ 音频资源：             ~8MB
  └─ 其他（配置/字体等）：  ~3MB
  ─────────────────────────────
  合计：                    ~48MB（预留 2MB 余量）
```

### 6.3 热更新方案

```
Cocos Creator 原生支持 Asset Bundle 热更新。

策略：
  • 核心代码和主场景资源打进初始包
  • 活动内容（限时涂装、节日资源）通过热更新下发
  • 热更新包每次 < 5MB（避免用户流量焦虑）
  • 可选择 Wi-Fi 下自动下载，流量下提示
```

## 七、决策记录（ADR）

```
ADR-001：选择 Cocos Creator 而非 Unity
  状态：✅ 已接受
  理由：包体小 20MB、2D 性能更好、低端机覆盖更广、可发小游戏
  如果后续发现：团队无法上手 Cocos → 回退到 Unity 备选方案

ADR-002：TypeScript 而非 C#
  状态：✅ 已接受
  理由：Cocos Creator 原生支持 TS，TS 在前端/Node 生态通用
  如果后续发现：TS 性能瓶颈 → 热点路径改用 JS 或 AssemblyScript

ADR-003：单向数据流 + EventBus 解耦
  状态：✅ 已接受
  理由：模块间不直接依赖，方便单模块测试和替换
  如果后续发现：事件数量过多导致调试困难 → 引入状态管理中间件

ADR-004：单场景切换而非多场景堆叠
  状态：✅ 已接受
  理由：游戏只有 3 个场景，不需要复杂的场景栈管理
  如果后续发现：需要复杂场景过渡 → 使用 Cocos Scene 系统

ADR-005：无物理引擎
  状态：✅ 已接受
  理由：本游戏不需要碰撞检测、刚体模拟等物理功能
  所有动画使用 Tween 实现
```
